Deploy Audit AI v7 ke AWS untuk Uji Coba

Migration guide dari Fly.io + Vercel ke AWS. Sistem sudah Docker-ready — tidak butuh refactor mayor.

Disusun: 7 Juni 2026 · target audience: tim ops Inspektorat II · pre-implementation review

1 · Konteks & Mengapa AWS

Setup existing:

  • Fly.io backend FastAPI (Singapore region, Docker, volume 3 GB /data)
  • Fly Postgres shared-cpu-1x
  • Vercel frontend Next.js (auto-deploy dari Git)

Pertimbangan pindah ke AWS:

  • Compliance internal Komdigi — AWS Indonesia region (ap-southeast-3 Jakarta) tersedia; kebijakan data dapat tertib di 1 region nasional.
  • Konsolidasi infra Inspektorat — kalau Itjen sudah punya AWS account lain, konsolidasi memudahkan audit & billing.
  • Latensi — Jakarta region lebih dekat ke auditor (Fly.io Singapore juga oke, tapi Jakarta lebih cepat).
  • Service catalog lebih luas — RDS, Aurora, EFS, ECS, App Runner, Amplify, CloudWatch, Secrets Manager.
Catatan penting sebelum deploy:
  • Egress LLM — backend spawn claude CLI subprocess yg connect ke api.anthropic.com. Pastikan VPC Security Group / NACL izinkan outbound HTTPS ke endpoint Anthropic. Tidak ada VPC endpoint resmi Anthropic — egress lewat NAT Gateway.
  • Data sensitif audit — temuan/KKP/LHR menyimpan informasi Satker. Pastikan kebijakan internal mengizinkan pemrosesan via AWS (region Jakarta) + egress ke Anthropic.
  • Setup Fly.io existing tetap jalan — migrasi bisa paralel (blue/green): AWS jadi sandbox, Fly.io tetap utk produksi sementara. Cut-over setelah uji coba sukses.

3 · Tiga Arsitektur Opsi

Opsi A — App Runner + Amplify + RDS + S3 Auto-scale

Cost: ~$30-60/bulan trial · Effort: Sedang (perlu refactor data ke S3)

Komponen:

  • Backend — App Runner deploy dari ECR image. Auto-scale 0-N instance, mirip Fly.io behavior.
  • Frontend — Amplify Hosting (auto-deploy dari Git, sertifikat HTTPS otomatis).
  • DB — RDS Postgres db.t4g.micro atau Aurora Serverless v2 (0.5 ACU min).
  • File storage — S3 (App Runner tidak support EFS mount). Perlu refactor: APP_DATA_DIR jadi prefix S3 (mis. s3://audit-ai-v7/data/penugasan/<kode>/).
  • Secrets — Secrets Manager, di-inject ke App Runner via runtime env.

Pros: auto-scale ke 0 saat idle (hemat); managed sepenuhnya.

Cons: refactor data layer butuh kerja (semua read/write file → S3 SDK). V6 scripts perlu cara akses S3 (atau tmpfs local + sync to S3 setelah batch selesai).

Opsi B — ECS Fargate + Amplify + RDS + EFS Code as-is

Cost: ~$80-150/bulan trial · Effort: Rendah (code existing langsung jalan)

Komponen:

  • Backend — ECS Fargate task dgn EFS volume mount → APP_DATA_DIR=/data. Behind ALB dgn ACM cert.
  • Frontend — Amplify Hosting.
  • DB — RDS Postgres db.t4g.micro.
  • File storage — EFS (POSIX-compatible). Code v7 + V6 tidak perlu diubah — sama dgn Fly Volume behavior.
  • Secrets — Secrets Manager, di-inject ke task definition.

Pros: drop-in replacement Fly.io. Tidak refactor. Production-grade (ALB + auto-recovery).

Cons: EFS & NAT Gateway = baseline cost lebih tinggi ($30/bulan NAT). Setup ALB + Target Group + SG sedikit lebih ribet dari App Runner.

Opsi C — EC2 single-server + docker-compose REKOMENDASI UJI COBA

Cost: ~$30-40/bulan trial · Effort: Paling rendah (~30-60 menit)

Komponen:

  • 1 EC2 t3.medium (2 vCPU + 4 GB RAM) di Jakarta region — jalankan docker-compose.yml existing langsung.
  • DB — Postgres di container (sederhana) atau RDS (production-grade). Trial bisa Postgres container; nanti migrate ke RDS.
  • File — EBS volume terlampir EC2 (gp3, 20 GB). Backup via snapshot harian.
  • Frontend — bisa di EC2 sama (port 3000 di-proxy nginx) atau Amplify Hosting terpisah.
  • HTTPS — nginx + ACM cert via Certbot/Let's Encrypt, atau pasang di belakang ALB dgn ACM cert.
  • Secrets — file .env di EC2 (dgn IAM role kalau ingin baca dari Secrets Manager).

Pros:

  • Paling cepat hidup. Setup sama dgn dev lokal (docker compose up -d).
  • Cost paling rendah. EC2 t3.medium on-demand ~$30/bulan; Reserved 1-year ~$18/bulan.
  • Mudah di-rebuild kalau salah konfigurasi — tinggal hapus EC2 + buat baru.
  • Cocok untuk 1-5 user concurrent uji coba 1-2 minggu.

Cons:

  • Single point of failure — kalau EC2 down, sistem mati. OK utk trial.
  • Scale terbatas — > 10 user concurrent perlu naik tier atau pindah ke B.
  • Update/maintenance manual (vs managed services di A/B).

Cocok kalau: uji coba terbatas 1-5 user · timeline pendek 1-4 minggu · belum yakin AWS jadi platform produksi.

Rekomendasi alur upgrade: mulai dgn Opsi C untuk uji coba awal. Setelah validasi konsep + auditor adopt, naikkan ke Opsi B (ECS+EFS+RDS) untuk produksi. Lewati Opsi A kecuali memang ingin investasi refactor data layer.

4 · Step-by-Step Opsi C (Rekomendasi Uji Coba)

4.1 Prasyarat

  • AWS account dgn permission EC2, RDS (opsional), Route 53 (opsional), ACM (opsional).
  • Domain (opsional, mis. audit-trial.komdigi.go.id) — bisa pakai IP publik EC2 utk awal.
  • ANTHROPIC_API_KEY ready.
  • SSH key pair AWS region Jakarta (ap-southeast-3).

4.2 Provision (lewat AWS Console atau CLI)

  1. VPC default — pakai yang sudah ada.
  2. Security Group baru audit-trial-sg:
    • Inbound: 22 (SSH dari IP kamu), 80 (HTTP), 443 (HTTPS), 3000 (frontend opsional), 8000 (backend opsional).
    • Outbound: all (default).
  3. EBS volume gp3 20 GB di-attach ke EC2 sebagai /dev/xvdf. Format ext4, mount /data.
  4. EC2 instance: AMI Ubuntu 22.04 LTS atau Amazon Linux 2023, type t3.medium, root 30 GB, SG = audit-trial-sg, key pair siap.
  5. Elastic IP (opsional, agar IP tidak berubah saat restart).
  6. Route 53 record A → Elastic IP (kalau pakai domain).
  7. ACM cert utk domain (opsional, untuk HTTPS via nginx).

4.3 Bootstrap server

SSH ke EC2 lalu:

# 1. Install Docker + git
sudo apt update && sudo apt install -y docker.io docker-compose-plugin git nginx certbot python3-certbot-nginx
sudo systemctl enable docker && sudo usermod -aG docker $USER
# logout/login agar group berlaku

# 2. Clone repo
cd /opt
sudo git clone https://github.com/irfansihab/audit-system-v7.git audit
sudo chown -R $USER:$USER audit
cd audit

# 3. Setup .env
cp .env.example .env
# edit .env:
#   ANTHROPIC_API_KEY=sk-ant-...
#   APP_SECRET_KEY=$(python3 -c 'import secrets; print(secrets.token_hex(32))')
#   APP_DATA_DIR=/data
#   APP_WIKI_PATH=/opt/audit/knowledge/wiki
#   ... (lihat .env.example)

# 4. Bind /data ke EBS volume (sudah di-mount di /data via /etc/fstab)
sudo mkdir -p /data && sudo chown $USER:$USER /data

# 5. Start docker compose
docker compose up -d db
sleep 5
cd backend && PYTHONPATH=. python3 -m app.init_db
cd ..
docker compose up -d backend frontend

# 6. nginx reverse proxy + HTTPS (kalau domain)
sudo tee /etc/nginx/sites-available/audit <<EOF
server {
  listen 80;
  server_name audit-trial.komdigi.go.id;
  location / { proxy_pass http://localhost:3000; }
  location /api/ { proxy_pass http://localhost:8000/; }
}
EOF
sudo ln -s /etc/nginx/sites-available/audit /etc/nginx/sites-enabled/
sudo nginx -t && sudo systemctl reload nginx
sudo certbot --nginx -d audit-trial.komdigi.go.id

# 7. Verifikasi
curl http://localhost:8000/health
curl http://localhost:3000/

4.4 Backup & recovery

  • EBS snapshot harian (Data Lifecycle Manager) — restore = bikin EC2 baru + attach snapshot.
  • Postgres dump mingguan ke S3: docker exec audit-db-1 pg_dump -U audit audit_v7 | gzip > /data/backup/audit_$(date +%F).sql.gz + aws s3 cp ... s3://audit-backups/.

4.5 Update / redeploy

cd /opt/audit
git pull
docker compose up -d --build
# Migrate DB schema kalau ada perubahan model
cd backend && PYTHONPATH=. python3 -m app.init_db

5 · Step-by-Step Opsi B (Production-Grade)

High-level — detail Terraform/CloudFormation di commit terpisah saat siap.

5.1 Infrastruktur

  1. VPC custom 2 public subnet + 2 private subnet (untuk RDS + EFS).
  2. NAT Gateway 1 buah (utk outbound egress ke api.anthropic.com).
  3. ECR repository audit-ai-v7 — push image dari laptop atau CI.
  4. EFS file system + access point (POSIX UID 1000), mount target di setiap AZ.
  5. RDS Postgres db.t4g.micro di private subnet, SG izinkan dari app SG.
  6. Secrets Manager: audit/anthropic_api_key, audit/app_secret_key, audit/db_password.
  7. ECS cluster Fargate, task definition baca secret + mount EFS, port 8000.
  8. ALB listener 443 (ACM cert) → target group ECS, health check /health.
  9. Amplify Hosting connect ke GitHub → auto-deploy frontend (env NEXT_PUBLIC_API_BASE = https://audit-trial.komdigi.go.id).
  10. Route 53 A record → ALB.

5.2 Initial deploy

  1. Build & push image: 
    aws ecr get-login-password | docker login --username AWS --password-stdin <account>.dkr.ecr.ap-southeast-3.amazonaws.com
cd backend
docker build -t audit-ai-v7 .
docker tag audit-ai-v7:latest <account>.dkr.ecr.ap-southeast-3.amazonaws.com/audit-ai-v7:latest
docker push <account>.dkr.ecr.ap-southeast-3.amazonaws.com/audit-ai-v7:latest
  2. ECS task definition: container image dari ECR, env dari Secrets Manager, mount EFS ke /data, command default (uvicorn).
  3. Run one-off task python -m app.init_db untuk migrate schema awal.
  4. Service desired-count 1, health check /health.
  5. Amplify Hosting: connect Git → set env → trigger build.

9 · Keputusan Terbuka Sebelum Deploy

  1. AWS Account — pakai account Itjen yg sudah ada, atau buat baru utk audit-ai trial?
  2. Compliance — apakah egress ke api.anthropic.com (di luar AWS) sudah dapat klarifikasi dari infosec? Apakah data audit boleh di region Jakarta?
  3. Domain — pakai domain Komdigi (mis. audit-trial.komdigi.go.id) atau cukup IP publik EC2 / subdomain default Amplify?
  4. Dimensi uji coba — berapa user concurrent? Berapa lama? (Opsi C cukup 1-5 user/1-2 minggu; Opsi B untuk >5 user/>1 bulan.)
  5. Fly.io paralel — Fly.io tetap jalan sebagai produksi sementara, atau langsung cut-over ke AWS?
  6. Vercel migration — frontend tetap di Vercel, atau pindah Amplify Hosting bareng?
  7. Backup & DR — apakah cukup EBS snapshot harian, atau perlu DR cross-region (Surabaya = belum ada region, fallback Singapore)?
