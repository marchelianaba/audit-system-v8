Rencana — Format Laporan Non-KKSA

Memo Konsultansi & Evaluasi RB (tabel 4-dimensi) di luar paradigma KKSA reviu
Inspektorat II Komdigi · render sisi-app (V6 tetap read-only) · profil format per skill

1. Masalah

Semua skill kini me-render LHP lewat render_lhp.py V6 — placeholder-fill ber-paradigma "reviu" KKSA (Kondisi-Kriteria-Akibat-Rekomendasi + simpulan keyakinan). Beberapa jenis pengawasan TIDAK memakai KKSA (lihat knowledge/skills/panduan-format-umum/PANDUAN.md):

JenisFormat wajibElemen
KonsultansiMemo KonsultasiDasar hukum + Pertanyaan + Pendapat/Saran — tanpa temuan, tanpa keyakinan
Evaluasi RBTabel 4-dimensiPer komponen Renaksi: Ketepatan Pelaksanaan · Ketercapaian Output · Kualitas Pelaksanaan · Kesesuaian Waktu (Sesuai/Tidak Sesuai)
Reviu LKj (SAKIP) / SPIPTabel per-kriteriaHasil per kriteria; tanpa "Sebab" (parsial — bisa menyusul)
Audit / Eval Kinerja-MRKKSA penuh✓ sudah didukung (KKSA)
Kendala: render_lhp.py ada di backend/v6/ (read-only, tak boleh diubah) dan mapping-nya menulis narasi reviu (mis. "catatan reviu … paradigma Kondisi-Kriteria-Akibat-Rekomendasi"). Memo & RB butuh struktur & narasi berbeda → perlu renderer sisi-app.

2. Konsep: Profil Format per Skill

Tiap skill punya profil format laporan. Diturunkan folder-driven dari frontmatter SKILL.md (tambah field format_laporan: kksa|memo|rb-4dim|tabel-kriteria) atau peta default. Renderer memilih jalur sesuai profil.

ProfilSkillRendererSumber data
kksa (default)reviu-rka-kl, audit-*, evaluasi-kinerja/MR, pemantauan-*render_lhp.py V6 (sudah jalan)temuan.json + rekomendasi.json + context.md
memokonsultansi-umum, konsultasi-pengadaanapp: render_memo (python-docx)saran.json (pertanyaan→pendapat/saran) + context.md
rb-4dimevaluasi-reformasi-birokrasiapp: render_rb (python-docx)penilaian RB (4-dim per komponen) dari gate/LKE
Prinsip: V6 tetap read-only. Renderer baru (memo/rb) milik app (python-docx, mengisi skeleton knowledge/templates/_skeleton-lhp/template-lhp-[skill].docx yang sudah ada). Satu tool dispatcher render_report(skill, …) memilih jalur sesuai profil — menggantikan pemanggilan langsung render_lhp di prompt KT.

3. Fase

FASE 1  Profil format + dispatcher

  • format_registry.py: peta skill → profil (frontmatter format_laporan atau default; folder-driven).
  • Tool render_report(penugasan_folder, skill, judul, …) di lhr_tools: profil kksa → delegasi ke render_lhp (sekarang); profil lain → renderer app. Prompt KT pakai render_report.
  • Tidak mengubah perilaku skill KKSA (regresi nol).

FASE 2  Memo Konsultansi

  • Skema _LHP/saran.json: [{pertanyaan, dasar_hukum[], pendapat, saran}] (AT/KT susun, bukan temuan KKSA).
  • Template template-lhp-konsultasi-pengadaan.docx + tambah placeholder {{MEMO_BLOK}} (loop pertanyaan→pendapat→saran). Konsultansi-umum: pakai template generik-memo.
  • render_memo (app, python-docx): isi identitas + ekspansi blok memo; tanpa simpulan keyakinan/temuan. Bahasa "pendapat/saran, tidak mengikat".
  • Prompt: skill konsultansi → tulis saran.json via tool baru append_saran, lalu render_report.

FASE 3  Evaluasi RB (tabel 4-dimensi)

  • Skema penilaian RB: per komponen Renaksi × 4 dimensi (Sesuai/Tidak Sesuai) + catatan. Sumber: hasil gate bertahap RB (sudah ada) / LKE-RB.
  • Template template-lhp-evaluasi-reformasi-birokrasi.docx + placeholder {{RB_TABEL_4DIM}} (ekspansi jadi tabel komponen × 4 dimensi) + {{RB_ANALISIS_DAMPAK}}.
  • render_rb (app): bangun tabel 4-dimensi + analisis dampak + AoI; tanpa KKSA.
  • Integrasi dengan gate RB: skor dimensi yang dikunci tiap gate jadi sumber tabel.

4. QC SAIPI

QC SAIPI saat ini berbasis temuan KKSA. Untuk profil memo/rb: QC profil-aware — lewati cek elemen KKSA yang tidak relevan (Memo: tak ada "akibat/keyakinan"; RB: format tabel). Tambah cek minimal khusus (Memo: ada dasar hukum & pendapat; RB: tiap komponen terisi 4 dimensi). Bisa menyusul di akhir.

5. Risiko & non-goals

  • V6 read-only dijaga — renderer memo/rb murni app-side.
  • Skeleton memo/RB yang ada saat ini masih bertemplate KKSA-reviu → perlu ditambah placeholder khusus (atau dibuat ulang ramping). Pekerjaan template manual oleh auditor bisa menyusul.
  • Bukan tujuan sekarang: Reviu LKj/SPIP tabel-kriteria (profil tabel-kriteria) — fase lanjutan; sementara tetap KKSA/generik.
  • Tidak memalsukan dasar hukum/angka — narasi dari dokumen + wiki regulasi.

Disusun untuk persetujuan sebelum implementasi. Urutan disarankan: Fase 1 → 2 (Memo, lebih sederhana) → 3 (RB, terikat gate).
