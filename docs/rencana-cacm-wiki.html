Rencana Implementasi — CACM & Wiki

Audit AI v7 · Inspektorat II Komdigi
Revisi 22 Mei 2026 · Status: rencana (belum ada kode) · Pendamping: ROADMAP.md, README.md

Dua fitur saling mengisi dalam satu loop: CACM menjadi sumber proaktif penugasan, dan Wiki menjadi memori organisasi yang dibaca agen saat analisis lalu diperbarui setelah penugasan selesai.

Prinsip dasar

Vault = sumber eksternalllm-wiki/ tetap repo terpisah (Karpathy method via Claude+Obsidian). Sistem audit terhubung sebagai referensi & pengusul update — tidak meng-vendor vault.
Human-in-the-loopSetiap tulisan ke wiki diusulkan dulu untuk disetujui auditor, mengikuti aturan sitasi (source: …) — bukan auto-commit.
Agen tetap hardenedAgen in-app tools=[] (tak bisa browser/shell). Scraping CACM tetap di luar app; app menerima hasilnya.
Prototype dulu, API kemudianCACM diuji lewat scrape SIRUP manual via Claude. Bila terbukti, baru sambung ke API/DB CACM organisasi.
🔁 Loop tertutup: CACM ➝ usulan penugasan ➝ reviu selesai ➝ temuan memperbarui Wiki ➝ Wiki memperkaya konteks penugasan & interpretasi sinyal CACM berikutnya.
✅ Status implementasi (per 25 Mei 2026) W1 (baca vault) — SELESAI: search_wiki/get_wiki_page + panel "Cari Wiki".
CACM C1a/C1b/C2 — SELESAI, dan arsitektur CACM berkembang dari sketsa "scrape manual" di bawah: tim menyerahkan layanan EWS SIRUP mandiri (agent Node/TS, crawl + 9 skenario + webhook HMAC + REST API). Jadi v7 = penerima: POST /cacm/ews-webhook (HMAC) + POST /cacm/sync (pull) + auto-usulan dari finding MERAH (anti-duplikat) + badge notifikasi. Ingest offline (/ingest-sample) untuk demo. Sisa: Wiki W2 (promosi pattern) & W3 (tulis-balik). Detail terkini: ROADMAP.md §13.
Bagian C1/C2 di bawah adalah rencana awal (sebelum delivery tim) — disimpan sebagai konteks historis.

A. Wiki — Integrasi Dua Arah knowledge

Kondisi sekarang

  • Baca (sebagian): agen memakai list/get_temuan_patterns + list/get_konteks atas wiki/ proyek (subset terkurasi).
  • Tulis-balik: belum ada. submit_feedback menghasilkan pattern_suggestions yang hanya mengendap di _FEEDBACK-AGEN/ + tampil di /feedback.
  • Vault penuh (llm-wiki/): 179 catatan + index.md + log.md, hasil ingest Karpathy dari dokumen resmi non-rahasia — belum tersentuh agen.

Baca: agen menarik konteks (profil auditi, riwayat BPK, vendor, regulasi). Tulis: usulan catatan terkurasi yang disetujui auditor lalu di-apply.

W1 Baca penuh vault PRIORITAS

  • Config APP_VAULT_PATH (default = llm-wiki/), read-only.
  • Tool baru: search_wiki(query) (index-driven + keyword atas index.md & isi) dan get_wiki_page(name).
  • Prompt AT/KT: langkah opsional "cari konteks auditi/vendor/BPK di vault" sebelum susun temuan.
  • UI: panel "Cari Wiki" di tab Knowledge (search ➝ daftar ➝ preview).

W2 Pemantauan & promosi pattern

  • GET /knowledge/pattern-monitor — agregasi pattern_suggestions lintas penugasan (frekuensi, contoh, status).
  • POST /knowledge/patterns — promosikan usulan disetujui menjadi file pattern di wiki/temuan-patterns/{skill}/.
  • UI: tabel usulan ➝ tombol "Promosikan" / "Tolak".

W3 Tulis-balik penugasan ➝ vault (tutup loop)

  • Saat penugasan LHP_DONE: hasilkan draft catatan pengawasan-{kode}.md (Summary, Sources=LHP/KKP, temuan+rekomendasi, [[wikilink]] ke auditi/regulasi/vendor) + delta index.md & log.md.
  • Review & apply: draft tampil di tab Knowledge (status USULAN). Auditor approve ➝ opsi A export .md untuk di-apply via Obsidian (rekomendasi) atau opsi B app menulis langsung ke vault.
  • Model WikiProposal untuk melacak usulan.

B. CACM — Prototype Scrape SIRUP Manual monitoring

Kondisi sekarang

  • Sistem CACM nyata (aggregator MonSAKTI/SIRUP/KRISNA) — sudah tersambung banyak API, tapi baru jadi database, belum dimanfaatkan.
  • Skill ews-cacm-komdigi sudah bisa scrape CACM via Claude-in-Chrome + evaluasi threshold (rasio RUP↔DIPA, konsentrasi vendor >30%, PDN <40%, kontrak lewat tempo, RO realisasi 0).
  • Di app: tab /cacm masih scaffold; PenugasanStatus belum punya USULAN_CACM.

Scraping di luar app (agen in-app tak punya browser). App = penampung sinyal + evaluator + alur usulan. Fase berikut tinggal ganti sumber sinyal: manual ➝ API CACM.

C1 Penampung sinyal + usulan penugasan PROTOTYPE

  • Model: PenugasanStatus + USULAN_CACM; tabel CacmSignal (sumber=SIRUP, satker, periode, payload, severity, status).
  • Backend routes/cacm.py: POST /cacm/signals (terima hasil scrape, jalankan evaluator), GET /cacm/signals, POST /cacm/signals/{id}/promote (buat penugasan USULAN_CACM ter-prefill), /dismiss.
  • Evaluator app/cacm_rules.py — port subset ews-rules untuk SIRUP ➝ severity + alasan.
  • UI /cacm: ingest sinyal (paste/upload), tabel sinyal + severity, tombol "Jadikan usulan" / "Abaikan"; badge "Usulan CACM" di list penugasan.

C2 Konektor API CACM SETELAH PROTOTYPE

  • app/cacm_connector.py — klien ke API/DB CACM organisasi, menggantikan intake manual.
  • POST /cacm/sync terjadwal (cron / skill schedule) ➝ tarik SIRUP harian ➝ sinyal otomatis.
  • (Opsional) push ringkasan hasil reviu balik ke anotasi CACM.

D. Urutan yang Disarankan

  1. W1 — Baca vault. Nilai tinggi, risiko rendah; langsung memperkaya agen.
  2. C1 — Sinyal SIRUP + usulan penugasan. Prototype CACM end-to-end, feed manual.
  3. W2 — Promosi pattern. Manfaatkan feedback yang sudah ada.
  4. W3 — Tulis-balik penugasan ➝ wiki. Menutup loop pengetahuan.
  5. C2 — Konektor API CACM. Setelah prototype terbukti bermanfaat.

E. Keputusan yang Dibutuhkan
