Kontrak API — INTEGRAL (Audit AI v8) ⇄ SIMWAS v2

v7 = mesin AI headless · SIMWAS = orkestrator, system of record, dan UI

Disusun 11 Juni 2026 · Spesifikasi mesin: docs/openapi-integral-v8.yaml

Ringkasan satu paragraf. SIMWAS v2 memegang seluruh siklus pengawasan: data, alur kerja, template, dan tampilan yang dilihat auditor. Audit AI v8 dipanggil di tahap-tahap tertentu untuk pekerjaan AI berat (digest dokumen, produksi temuan, draft laporan) lalu mengembalikan hasil terstruktur (JSON) dan dokumen jadi (.docx). v7 tidak punya tampilan produksi — tim SIMWAS membangun layarnya di Laravel/Blade dan memanggil API ini.

2. Topologi penempatan (rekomendasi)

Karena SIMWAS sudah berjalan di servernya sendiri, cara termudah memulai: v7 ditaruh sebagai stack Docker terpisah di server yang sama, hanya bisa diakses SIMWAS lewat jaringan lokal — tidak dibuka ke publik. Pindah ke server sendiri belakangan cukup ganti alamat, tanpa ubah kode.

3. Autentikasi — tiga kanal

Kanal 1 User (SSO)

SIMWAS menerbitkan JWT bertanda-tangan; v7 memverifikasi lewat JWKS publik SIMWAS. Klaim minimal: sub, role (AT/KT/PT/PM), nama, exp. Dikirim Authorization: Bearer <jwt>.

Menggantikan login prototype v7 dan memperbaiki akar masalah token kedaluwarsa (error 401).

Kanal 2 Server-ke-server

Panggilan SIMWAS→v7 tanpa konteks user (mis. sync ST) memakai header X-SIMWAS-API-KEY. Pola ini sudah ada presedennya di v7 (X-API-Key untuk agen CACM).

Kanal 3 Webhook (v7→SIMWAS)

Saat analisis selesai, v7 memanggil balik URL SIMWAS. Body ditandatangani HMAC-SHA256 (X-SIMWAS-Signature: sha256=<hex>) — pola yang sudah dipakai v7 untuk webhook CACM.

Kenapa webhook, bukan polling?

Analisis agen AI berjalan lama (puluhan detik–menit). SIMWAS jangan menunggu di dalam satu request. Pola: SIMWAS memicu → dapat run_id → menerima webhook saat selesai.

4. Urutan panggilan per-tahap

Pemetaan 7 tahapan INTEGRAL ke panggilan API v7. Tahap yang diberi tanda live sudah berjalan & terverifikasi E2E pada penugasan uji.

  1. ST → penugasan proposed
    POST /api/simwas/st/sync — SIMWAS push Surat Tugas; v7 buat/mirror penugasan + isi pelaksana & periode. (Sekarang penugasan dibuat via POST /penugasan.)
  2. Kartu Penugasan (PT) live
    PUT /penugasan/{id}/kp-md — simpan KP + sasaran (auto-sync ke PKP).
  3. PKP (KT) live / manual
    PUT /penugasan/{id}/sasaran-assignment, atau tarik dari SIMWAS lewat POST …/sasaran/sync-from-simwas (source=manual aktif; api menunggu kontrak).
  4. Upload & digest dokumen (AT) live
    POST /dokumen (multipart) — auto-digest di background (TOR/RAB deterministik + fallback AI).
  5. Generate konteks (AT) live
    GET /agen/anggota_tim/stream?prompt=[MODE:CONTEXT]… (SSE) — rakit context.md dari 4 sumber.
  6. Analisis → temuan (AT) live
    GET /agen/anggota_tim/stream (SSE) → tulis _KKP/temuan.json. Lihat …/temuan-review.
  7. HITL: review temuan live
    POST …/temuan-review/{tid}/approve|reject, PUT …/edit, POST …/bulk-approve.
  8. Draft LHP (KT) live
    GET /agen/ketua_tim/stream (SSE) → hasil .docx + QC SAIPI.
  9. Reviu konsep LHP (PT/PM) live
    POST /penugasan/{id}/lhp-review — APPROVED / NEEDS_REVISION (+catatan wajib bila revisi).
  10. Ambil hasil live
    GET …/files → GET …/files/download?path=_LHP/LHP-SUBSTANSI.docx.

5. Format hasil yang v7 kembalikan

Temuan (KKP) — JSON

Dari GET …/temuan-review atau preview _KKP/temuan.json. Tiap temuan: id_temuan, judul, kondisi, kriteria, akibat, rekomendasi, referensi_sasaran, status. Dipetakan ke kolom KKSA SIMWAS.

Draft LHP — dokumen .docx

Diunduh lewat …/files/download. Sudah dirender V6 sesuai format laporan + lulus QC SAIPI (REN-001…007). SIMWAS cukup menyajikan/lampirkan.

Hasil QC SAIPI — JSON

_KKP/kkp-qc-result.json, _LHP/lhp-qc-result.json: jumlah OK/KRITIS per aturan, untuk gerbang mutu sebelum tahap lanjut.

Status penugasan — enum

DRAFT → INGESTING → KKP_IN_PROGRESS → KKP_DONE → LHP_IN_PROGRESS → LHP_DONE. Diturunkan v7 dari artefak; SIMWAS bisa menampilkan progres tanpa menyimpan ulang.

7. Pembagian pekerjaan

Disediakan tim SIMWAS
  • Penerbit SSO + JWKS publik (verifikasi token) & format klaim
  • Nilai X-SIMWAS-API-KEY untuk panggilan server-ke-server
  • Endpoint penerima webhook + secret HMAC
  • Pemetaan jenis pengawasan SIMWAS → slug skill v7
  • Sumber daya server (RAM/CPU) + izin egress ke api.anthropic.com
  • Layar Blade yang memanggil endpoint v7
Dibangun / disiapkan v7
  • Verifikasi JWT SIMWAS via JWKS (ganti login prototype)
  • Middleware X-SIMWAS-API-KEY
  • Endpoint POST /api/simwas/st/sync (isi pelaksana + periode)
  • Implementasi source=api pada sync PKP (kini 501)
  • Pemanggil webhook keluar saat KKP/LHP selesai
  • Berkas OpenAPI final + contoh payload (Postman collection)
Bisa dikerjakan sekarang tanpa menunggu tim SIMWAS. Endpoint st/sync, middleware API-key, dan pemanggil webhook dapat dibangun & diuji mock-first memakai payload contoh. Begitu tim SIMWAS menyediakan JWKS, API-key, dan URL webhook, tinggal disambungkan.

8. Keputusan yang masih terbuka

  • Egress LLM ke Anthropic. v7 memanggil api.anthropic.com untuk analisis. Untuk data audit sensitif di PDN/PDNS, perlu keputusan kepatuhan: zero-retention/enterprise, gateway redaksi, atau model on-prem.
  • Format identitas SSO. Klaim & masa berlaku token SIMWAS; apakah ada konsep "role aktif" yang bisa berganti.
  • Pemetaan jenis ↔ skill. Daftar resmi jenis pengawasan SIMWAS dipetakan ke 17 skill v7.
  • Validasi konten template KP/PKP oleh auditor senior (di luar lingkup koding).
